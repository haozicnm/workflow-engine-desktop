## #1 · Parser ↔ Executor 契约断裂

### 验证结果：确认存在

**Parser 定义**（唯一权威来源）[`src-tauri/src/engine/parser.rs:20`]：

```rust
pub const CONTAINER_TYPES: &[&str] = &["browser", "excel", "word", "logic", "file"];
```

**Executor 注册**（必须手动同步）[`src-tauri/src/engine/executor.rs:88-94`]：

```rust
register_containers!(executors,
    "browser" => crate::nodes::browser_container::BrowserContainerNode,
    "excel"   => crate::nodes::excel_container::ExcelContainerNode,
    "word"    => crate::nodes::word_container::WordContainerNode,
    "logic"   => crate::nodes::condition::ConditionNode,
    "file"    => crate::nodes::file_container::FileContainerNode,
);
```

### 证据链

① 二者**当前一致**（5 个类型一一对应），但一致性靠人肉维护。  
② 存在测试 `container_types_match_registrations` [`src-tauri/src/engine/executor.rs:196-218`]，但它**独立创建 HashMap 并重新调用 `register_containers!`，不测试真实的 `StepExecutor::new()` 注册结果**：

```rust
// 测试用独立的 HashMap —— 不是 StepExecutor::new() 的 executors
let mut executors: HashMap<String, Box<dyn NodeExecutor>> = HashMap::new();
register_containers!(executors,    // <-- 重复声明，非真实 executor
    "browser" => ...,
    ...
);
```

③ 另一个测试 `all_registered_types_can_be_instantiated` 确实检查了真实 `StepExecutor`，但它有一个**硬编码的 `required` 列表** [`src-tauri/src/engine/executor.rs:220-231`]：

```rust
let required = &[
    ..., "browser_container", "excel_container", "word_container",
    "file_container", "logic_container", ...
];
```

**这意味着新增容器类型时，需要在 3 处同步添加**：`CONTAINER_TYPES` → `register_containers!` → `required` 列表。漏改任何一处都不会在编译期报错。

### 风险

历史上已触发：集成测试中 `file` 容器漏加导致运行时失败。

### 修复建议

对应 ROADMAP A1 + A5：

```rust
// build.rs 生成测试：直接从真实 StepExecutor::new() 读取 executors，
// 遍历 parser::CONTAINER_TYPES，断言每个 "{type}_container" 都已注册。
// 删除测试内重复的 register_containers! 调用。
```

**具体步骤**：
1. 在 `StepExecutor` 上暴露一个 `pub fn registered_types(&self) -> Vec<&str>` 方法
2. 在 `build.rs` 或测试中：遍历 `CONTAINER_TYPES`，对每个类型断言 `executor.registered_types().contains(&format!("{t}_container"))`
3. 反向检查：遍历 `registered_types()` 中所有 `_container` 后缀的类型，断言 `CONTAINER_TYPES` 包含其基础名
4. 删除现有的重复 `register_containers!` 测试代码

**预期效果**：新增容器时只需改 `CONTAINER_TYPES` + `register_containers!`（两处），编译器 + 测试自动校验一致性。

`deepMerge` 提取到 `src/lib/utils.ts`（该文件已存在，目前只有 `cn()` 函数）。

**具体步骤**：

1. 将 `deepMerge` 移至 `src/lib/utils.ts`
2. 将 `validateRefs` + `normalizeSteps` 提取为 `src/composables/useWorkflowValidate.ts`
3. 将 `loadWorkflow` / `saveWorkflow` / `deleteWorkflow` / `cloneWorkflow` 提取为 `src/stores/workflowPersistence.ts`

## #4 · Rust 生产代码 `unwrap()` 使用

### 验证结果：确认存在，但严重度低于初步判断

**实际数量**：~10 处，均位于低风险场景：

| 文件 | 行号 | 代码 | 风险 |
|------|------|------|------|
| `cli.rs` | 170, 292, 379, 397, 685, 789 | `serde_json::to_string_pretty(&json!(...)).unwrap()` | 低 — serde 对已知结构体不会失败 |
| `cli.rs` | 397 | `serde_json::to_string_pretty(&manifest).unwrap()` | 低 |
| `ipc.rs` | 165 | `serde_json::to_string(&err).unwrap()` | 中 — IPC 错误路径 |
| `ipc.rs` | 296 | `serde_json::to_string(&ack).unwrap()` | 中 — IPC 路径 |
| `browser_container.rs` | 528 | `v.as_array().unwrap()` | 低 — 前置 `v.is_array()` 守卫 |

**不在生产代码中的**（均为 `#[cfg(test)]` 块内，可接受）：
- `executor.rs:192` — test helper `Runtime::new().unwrap()`
- `executor.rs:316` — test assertion 后的 `result.unwrap()`
- `parser.rs` 多处 — test 代码中 `parse_workflow(...).unwrap()`

### 修复建议

```rust
// 方案 A：用 expect 提供上下文
serde_json::to_string_pretty(&json!(...))
    .expect("CLI JSON 序列化不应失败")

// 方案 B（IPC 路径推荐）：返回错误而非 panic
let json = serde_json::to_string(&ack)
    .map_err(|e| anyhow::anyhow!("序列化 ack 失败: {}", e))?;

// 方案 C（browser_container）：用 and_then 链消除 unwrap
let file_paths: Vec<String> = action.config.get("file_paths")
    .and_then(|v| v.as_array())
    .map(|arr| arr.iter().filter_map(|x| x.as_str().map(|s| s.to_string())).collect())
    .unwrap_or_default();
```

**优先级**：先修复 IPC 路径的 2 处（`ipc.rs:165` + `ipc.rs:296`），因为它们在运行时错误路径上，panic 会杀死整个 IPC 服务器。

**证据** [`src-tauri/Cargo.toml:43-48`]：


- 第一段：`"键鼠模拟 + 屏幕截图"`

